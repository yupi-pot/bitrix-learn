## Принципы работы со списком

1. **Фазируй запросы.** Не проси всё сразу — клиент утонет. Phase 0 — к старту разработки, Phase 1 — в первые 2–3 недели, Phase 2 — под конкретные фичи, Phase 3 — к запуску.
2. **Фиксируй письменно.** Устные договорённости исчезают. Переписка, тикеты, совместные доки.
3. **Непослученное — это риск.** Раз в неделю на статусе: «X не получено, если не придёт до Y — сдвигаемся на Z».
4. **Не делай work-around на неполученных данных.** Лучше эскалировать сейчас, чем переделывать треть проекта перед запуском.

## Технический контекст текущего сайта (обновлено 2026-04-22)

Результаты изучения `https://geropharm.ru/`:

| Параметр | Значение | Последствие для проекта |
|---|---|---|
| Движок | **Yii2 (PHP-фреймворк)** | ⚠️ Полная замена платформы, не апгрейд. Миграция через свой скрипт, не через штатные импорты Битрикса |
| Веб-сервер | Apache | Совместимо с Битриксом |
| SSL | Let's Encrypt | Для нового сайта тоже берём LE, платный не нужен |
| Объём в sitemap | **2 983 URL** | Значительная миграция контента — учесть в оценке |
| Английская версия | Уже есть на `/en` | Архитектурно склоняет к multisite-режиму в Битриксе |
| URL-структура | Плоская (`/название-статьи` в корне) | ~3000 301-редиректов при переходе на новую структуру |
| robots.txt | Пустой (`User-agent: *`) | Без спецправил — нормально |
| CSP | Разрешены Яндекс, Google, VK, YouTube, RuTube, FB | На новом сайте нужно настроить те же встраивания |

**Главный риск:** миграция контента заложена в КП одной строчкой. Реальность — 3000 URL из кастомной Yii2-схемы БД. Нужно пересогласовать объём работ на миграции в первые же недели.

- [x] ~~**Карта URL текущего сайта**~~ — получено: `sitemap.xml`, 2983 URL, плоская структура
- [ ] **Политика по английской версии** — в Yii эти записи отдельные или общие с флагом языка? От этого зависит multisite-решение в Битриксе
